# LLM Wiki 기술 안내

이 문서는 LLM Wiki의 내부 구조와 AI 기능을 기술 배경이 없는 사용자도 이해할 수 있도록 설명합니다. 간단한 프로젝트 소개와 실행 방법은 [README](./README.md)를 참고하세요.

## 시스템 개요

LLM Wiki는 하나의 Node.js 서버가 화면과 API를 함께 제공하는 웹 애플리케이션입니다. 문서 작성, 검색, AI 질문, 충돌 확인이 같은 서버에서 동작합니다.

```text
웹 브라우저
  ↓ 문서 작성·검색·질문
Hono 웹 서버
  ├─ 문서 처리 규칙
  ├─ OpenRouter AI
  ├─ 로컬 임베딩 모델
  └─ 메모리 또는 Markdown 파일 저장소
```

Hono는 요청을 받아 알맞은 기능으로 전달하는 가벼운 웹 서버 도구입니다. 화면은 별도 프런트엔드 프로젝트 없이 서버가 HTML, CSS, JavaScript를 한 번에 만들어 제공합니다.

## 주요 기술

| 영역 | 기술 | 역할 |
|---|---|---|
| 실행 환경 | Node.js | 서버에서 JavaScript와 TypeScript 실행 |
| 개발 언어 | TypeScript | 데이터 형태를 검사할 수 있는 JavaScript |
| 웹 서버 | Hono | 화면과 API 요청 처리 |
| 외부 AI | OpenRouter | 요약, 분류, 충돌 분석, 답변 생성 |
| 로컬 AI | Transformers.js | 문장을 의미 벡터로 변환 |
| 문서 형식 | Markdown·YAML | 본문과 관리 정보를 사람이 읽을 수 있는 파일로 저장 |
| 편집기 | EasyMDE | 브라우저 Markdown 편집기 |
| 화면 렌더링 | Marked·DOMPurify | Markdown 변환과 위험한 HTML 제거 |
| 테스트 | Vitest | 기능과 API 자동 검증 |

## 코드 구조

프로젝트는 업무 규칙과 외부 기술을 분리합니다. 이를 헥사고날 아키텍처라고 하며, 쉽게 말하면 데이터베이스나 AI 제공자를 바꿔도 핵심 문서 규칙을 최대한 유지하는 구조입니다.

```text
src/domain          문서, 제목, 상태처럼 가장 기본적인 규칙
src/application     저장, 검색, 충돌 확인처럼 사용자가 수행하는 작업
src/infrastructure  웹 서버, OpenRouter, 파일 저장 등 외부 기술 연결
tests               각 계층의 자동 검증
scripts             보조 실행 도구
```

`port`는 애플리케이션이 필요로 하는 기능의 약속이고, `adapter`는 그 약속을 실제 기술로 구현한 코드입니다. 예를 들어 문서 저장 약속은 동일하지만 메모리 저장과 파일 저장 구현을 선택할 수 있습니다.

## 문서 저장 과정

신규 문서는 다음 순서로 처리됩니다.

1. 같은 제목이나 태그가 많이 겹치는 기존 문서를 먼저 확인합니다.
2. AI가 문서를 요약하고 주제를 제안합니다.
3. 사용자가 제안된 주제를 확인합니다.
4. 관련 문서 제목을 찾아 문서 연결을 제안합니다.
5. 존재하지 않는 연결과 순환 구조를 검사합니다.
6. 문서와 목록용 요약 정보를 저장합니다.
7. 구조적 충돌을 양쪽 문서에 반영합니다.
8. 같은 주제의 문서와 의미론적 충돌을 검사합니다.

같은 제목은 내부적으로 같은 문서 식별자로 변환됩니다. 이 식별자는 제목을 소문자로 만들고 공백을 하이픈으로 바꾼 값입니다. 예를 들어 `배송 정책`은 내부에서 `배송-정책`으로 관리됩니다. 신규 저장 시 같은 식별자가 있으면 기존 문서를 덮어쓰지 않고 저장을 중단합니다.

## AI 기능

### 요약과 주제 분류

문서 제목과 본문을 OpenRouter에 전달해 한국어 요약, 주제 이름, 신뢰도를 받습니다. 제안된 주제가 기존 주제와 충분히 비슷하면 같은 그룹으로 묶고, 그렇지 않으면 새 그룹을 만듭니다.

주제별 최상위 문서가 없으면 빈 안내 문서를 자동으로 만들고 새 문서를 그 아래에 배치합니다. 부모와 자식 문서가 서로를 반복해서 가리키는 순환 구조는 저장 전에 차단합니다.

### 의미론적 충돌

의미론적 충돌은 표현이 다른 문장 사이의 실제 모순을 찾는 기능입니다. 같은 주제의 문서만 비교하며 육하원칙을 사용합니다.

- 누가, 언제, 어디서, 무엇을 통해 같은 상황인지 확인합니다.
- 양쪽의 어떻게가 동시에 참일 수 없는지 확인합니다.
- 왜는 서로 배타적인 원인을 명시한 경우에만 사용합니다.
- 양쪽 원문에 실제 근거 문장이 있는 결과만 저장합니다.
- 기준 시점과 비교 시점을 별도로 확인합니다.

단순한 설명 차이, 빠진 정보, 서로 다른 시점·환경·버전은 충돌로 취급하지 않습니다. 문서를 수정하면 의미론적 충돌을 다시 검사하고 상대 문서의 상태도 함께 갱신합니다.

### Ask AI

먼저 `index.json`의 문서 제목과 AI 요약에서 질문과 직접 일치하는 정보를 찾습니다. 일치도가 높으면 해당 결과를 바로 사용하고, 약하면 로컬 임베딩의 의미 유사도 결과와 합쳐 관련성이 높은 문서를 최대 3개 선택합니다. 이후 OpenRouter가 선택된 문서만 참고해 답변합니다.

관련 문서끼리 충돌하면 임의로 한쪽을 선택하지 않고 충돌 해결을 먼저 안내합니다.

## 로컬 임베딩 모델

임베딩은 문장의 의미를 숫자 목록으로 바꾸는 과정입니다. 의미가 비슷한 문장은 가까운 숫자 결과를 만들기 때문에 단어가 정확히 같지 않아도 관련 문서를 찾을 수 있습니다.

현재 로컬 모델은 `Xenova/multilingual-e5-small`입니다.

- 한국어를 포함한 여러 언어를 지원합니다.
- 모델을 처음 사용할 때 내려받아 메모리에 올립니다.
- 가볍게 만든 양자화 모델을 사용합니다. 양자화는 정확도 손실을 줄이면서 모델 크기와 메모리 사용량을 낮추는 기술입니다.
- 문장마다 384개의 숫자로 구성된 벡터를 생성합니다.
- 두 벡터의 방향이 얼마나 비슷한지 계산해 문서 순위를 정합니다.

OpenRouter를 사용하는 기능과 달리 임베딩 생성 자체에는 외부 AI API 키가 필요하지 않습니다. 기본 설정은 `EMBEDDING_PROVIDER=local`입니다.

### 현재 임베딩 제한

Ask AI의 제목·요약 검색 결과가 충분하지 않을 때 서버가 현재 `index.json`의 제목과 요약을 자동으로 벡터로 바꿉니다. E5 모델의 검색 방식에 맞춰 질문에는 `query:`, 문서에는 `passage:` 구분을 적용합니다.

실행 중 사용하는 벡터 저장소는 메모리 기반입니다. 서버를 재시작하면 저장된 벡터가 사라지지만, 다음 의미 검색 때 최신 문서 목록을 기준으로 다시 생성합니다. 현재 방식은 문서 수가 많아질수록 재생성 비용이 커질 수 있습니다.

대규모 문서 환경에서는 다음 확장이 필요합니다.

- 벡터를 파일이나 데이터베이스에 영구 저장
- 서버 시작 시 기존 벡터 불러오기
- 변경된 문서의 벡터만 선택적으로 갱신

## 문서 연결

본문에 기존 문서 제목이 일반 텍스트로 등장하면 `[[문서 식별자]]` 형태의 연결을 제안합니다. 이미 연결된 부분은 다시 변경하지 않으며, 긴 제목부터 처리해 짧은 제목이 일부만 먼저 연결되는 문제를 줄입니다.

연결 대상 문서가 없으면 끊어진 연결로 표시합니다. 자기 자신을 가리키는 연결은 제외합니다.

## 저장 방식

### 메모리 저장

기본 설정입니다. 설치 없이 바로 실행할 수 있지만 서버를 종료하면 모든 문서가 사라집니다. 기본 제한은 문서 100개, 전체 약 1MB입니다.

### 파일 저장

`USE_FILE_STORAGE=true`로 설정하면 `data/wiki`에 문서를 저장합니다.

- 각 문서: Markdown 파일
- 문서 본문 위의 YAML 영역: 상태, 주제, 태그, 연결, 충돌 정보
- `index.json`: 목록 조회에 사용하는 제목과 요약 정보

YAML은 사람이 읽을 수 있는 설정 형식입니다. 본문과 관리 정보를 한 파일에 함께 보관할 수 있습니다.

## 화면 구성

- Write: Markdown 문서 작성, 충돌 사전 확인, AI 요약과 주제 확인
- Browse: 검색과 필터, 문서 계층 탐색, 상세 조회와 수정
- Ask AI: 저장된 문서를 바탕으로 질문과 답변
- 알림: 의미론적 충돌 목록과 근거 확인

화면은 EasyMDE, Marked, DOMPurify를 CDN에서 불러옵니다. CDN에 연결할 수 없는 폐쇄망에서는 이 파일들을 프로젝트 내부에 포함해야 합니다.

## 환경 변수

| 이름 | 기본값 | 설명 |
|---|---|---|
| `OPENROUTER_API_KEY` | 없음 | 외부 AI 기능 사용에 필요한 키 |
| `OPENROUTER_MODEL` | `openai/gpt-4o-mini` | 요약과 기본 AI 모델 |
| `OPENROUTER_DOMAIN_MODEL` | 기본 모델 | 주제 분류 전용 모델 |
| `OPENROUTER_ASK_MODEL` | 기본 모델 | Ask AI 전용 모델 |
| `OPENROUTER_SEMANTIC_CONFLICT_MODEL` | 기본 모델 | 의미론적 충돌 전용 모델 |
| `EMBEDDING_PROVIDER` | `local` | 로컬 임베딩 사용 여부 |
| `USE_FILE_STORAGE` | `false` | 파일 저장 활성화 |
| `DATA_ROOT` | `./data` | 문서 저장 폴더 |
| `HOST` | `127.0.0.1` | 서버가 요청을 받는 주소 |
| `PORT` | `3000` | 서버 포트 |

## 실행과 검증

```bash
npm run dev        # TypeScript 빌드 후 서버 실행
npm test           # 전체 자동 테스트
npm run test:watch # 변경할 때마다 테스트
npm run typecheck  # TypeScript 형태 검사
```

테스트는 핵심 문서 규칙, 저장 방식, OpenRouter 응답 처리, HTTP API를 외부 서비스 없이 검증합니다. 실제 네트워크 서버가 열리는지도 별도 테스트합니다.

## 운영 시 주의사항

- 인증과 사용자 권한 기능이 없습니다. 공개 인터넷 서비스로 운영하려면 인증을 추가해야 합니다.
- 파일 저장에는 여러 사용자가 동시에 수정할 때의 잠금 기능이 없습니다.
- 파일 저장은 쓰기 가능한 장기 실행 서버에 적합합니다. Vercel 같은 서버리스 환경에서는 외부 데이터베이스나 객체 저장소가 필요합니다.
- 신규 문서의 의미론적 충돌 검사는 저장 응답 뒤에 실행되므로, 서버리스 환경에서는 작업 완료를 보장하는 별도 처리가 필요합니다.
- 수정 재검사에서 AI 요청 실패와 실제 충돌 없음이 모두 빈 결과로 표현될 수 있어 기존 충돌이 해제될 가능성이 있습니다. 두 상태를 분리하는 개선이 필요합니다.
- 검색과 충돌 검사는 문서 전체를 순회하므로 문서 수가 매우 많아지면 데이터베이스 검색과 작업 큐가 필요합니다.

## 용어 정리

| 용어 | 쉬운 설명 |
|---|---|
| API | 화면과 서버가 데이터를 주고받는 통로 |
| 도메인 | 비슷한 문서를 묶는 주제 그룹 |
| 문서 식별자 | 제목에서 만든 내부 문서 이름 |
| 임베딩 | 문장의 의미를 비교 가능한 숫자로 바꾸는 과정 |
| 벡터 | 임베딩 결과로 만들어진 숫자 목록 |
| 의미론적 검색 | 같은 단어보다 같은 의미를 우선하는 검색 |
| 의미론적 충돌 | 표현은 달라도 사실상 동시에 참일 수 없는 정보 |
| 구조적 충돌 | 같은 제목이나 지나치게 겹치는 태그처럼 규칙으로 찾는 문제 |
| 인메모리 | 디스크가 아니라 실행 중인 메모리에만 보관하는 방식 |
| 영속 저장 | 서버를 재시작해도 데이터가 남는 저장 방식 |
| CDN | 편집기 같은 화면 파일을 인터넷에서 받아오는 서비스 |
| YAML | 사람이 읽기 쉬운 설정 데이터 형식 |
