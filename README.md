# LLM Wiki

AI가 문서의 분류, 연결, 충돌 확인을 도와주는 웹 기반 지식 관리 도구입니다.

문서를 작성하면 AI가 내용을 요약하고 적절한 주제를 제안합니다. 기존 문서와 내용이 모순되는지도 확인해 여러 사람이 함께 관리하는 문서의 일관성을 높입니다.

## 프로젝트 목적

팀 문서는 사람이 늘어날수록 분류 기준이 달라지고, 비슷한 문서가 흩어지며, 서로 다른 정보가 함께 남기 쉽습니다. LLM Wiki는 문서를 저장하는 순간부터 AI가 정리와 검증을 도와 지식이 일관된 형태로 쌓이게 하는 것을 목표로 합니다.

## 해결하려는 문제

- **사람마다 다른 분류 기준:** AI가 문서 내용을 바탕으로 주제를 제안해 일관된 구조를 유지합니다.
- **흩어진 관련 문서:** 기존 문서와의 연결을 제안해 필요한 정보를 함께 탐색할 수 있게 합니다.
- **서로 모순되는 정보:** 같은 상황을 다르게 설명하는 문장을 찾아 근거와 함께 보여줍니다.

## 주요 기능

### 문서 작성과 정리

- Markdown으로 문서를 작성하고 수정합니다.
- AI가 문서를 요약하고 적절한 주제를 제안합니다.
- 저장 전에 제안된 분류를 확인하거나 변경할 수 있습니다.
- 관련 문서를 자동으로 연결합니다.

### 충돌 확인

- 같은 제목의 문서가 있으면 저장 전에 알려주고 기존 문서를 보여줍니다.
- AI가 같은 상황을 서로 다르게 설명하는 문서를 찾습니다.
- 충돌한 양쪽 문장과 판단 기준을 함께 보여줍니다.
- 문서를 수정하면 충돌 여부를 다시 확인합니다.

### 문서 탐색

- 문서 목록을 검색하고 주제별로 살펴볼 수 있습니다.
- 문서 안의 연결을 따라 관련 내용을 이어서 읽을 수 있습니다.
- 충돌이 발견된 문서는 알림과 경고로 확인할 수 있습니다.

### Ask AI

- 저장된 문서를 바탕으로 자연어 질문에 답합니다.
- 참고한 문서를 답변과 함께 보여줍니다.
- 서로 충돌하는 지식이 있으면 답변보다 충돌 해결을 먼저 안내합니다.

## 빠른 시작

Node.js가 설치된 환경에서 실행합니다.

```bash
npm install
npm run dev
```

AI 기능을 사용하려면 프로젝트 루트의 `.env` 파일에 OpenRouter API 키를 설정합니다.

```env
OPENROUTER_API_KEY=your-api-key
```

## 문서 보관

파일 저장 기능을 사용하려면 다음 환경 변수를 설정합니다.

```env
USE_FILE_STORAGE=true
DATA_ROOT=./data
```

작성한 문서는 `data/wiki` 폴더에 Markdown 파일로 보관됩니다. 서버를 다시 실행한 뒤에도 문서를 유지하려면 파일 저장 기능을 사용하세요.

## 개발 명령어

```bash
npm run dev        # 개발 서버 실행
npm test           # 테스트 실행
npm run typecheck  # 타입 검사
```

기술 구조와 AI 동작은 [기술 안내](./TECH_README.md), 이슈와 기능 변경 내용은 [`docs`](./docs)에서 확인할 수 있습니다.

## 회고
### 병목은 나인듯
 - PRD.md를 생성 후 개발되는 프로젝트가 어느샌가 부터 제 생각보다 기능이 많아 졌습니다.
  - 저는 Link 기능을 생각하지 않았습니다.
 - LLM이 가지고있는 컨텍스트 언어가 저에겐 없습니다.
  - 구조적 Conlfict를 설명할때 Slug에 대한 이야기를 하였으나, 전 Slug가 뭔지 모릅니다.
  - 구조적 Conlfict, 의미론적 Conflict가 무엇인지 저와 논의한 적이 없습니다.
### 병목 제거공략
 - 하네스 엔지니어링을 도입하여 새로운 기능과 컨텍스트 언어가 필요해진다면 저를 먼저 이해시키게 합니다.
   - 진행속도와 정확히 트레이드 오프 관계인데 이것이 올바른 방향인지 모르겠습니다. 
 - 하네스 엔지니어링을 도입하여 새로운 기능과 컨텍스트 언어가 필요해진다면 docs 폴더에 정리하게 합니다.
   - 이후 사용자가 해당 문서를 확인하면 되지만, 게으른 제가 확인할지 모르겠습니다.
### 헥사고날 아키텍처는 바이브코딩에 알맞는 것 같습니다.
 - 문서를 저장하는 기능을 개발할때 충돌 내용이 있는지 판단하는 의미론적 Conlfict 기능 동작원리를 LLM이 알 필요가 없습니다.
  - 실제로 LLM은 의미론적 Conflict 코드베이스를 읽지 않고 Port인 인터페이스 파일만 해석하면 됩니다.
