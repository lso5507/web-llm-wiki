# Release 0.1 - Semantic Search Implementation

## Overview

Release 0.1 introduces semantic search capabilities to the Ask AI feature, replacing the previous keyword-based document retrieval with embedding-based similarity search. This enables the system to understand meaning rather than just matching exact keywords.

## Problem Statement

### Before
- **Keyword-only matching**: Ask AI used simple token frequency to find documents
- **Poor recall**: Query "배송비용" failed to find documents titled "한국 배송" or "해외배송비용"
- **No semantic understanding**: Could not match synonyms, related terms, or different expressions of the same concept
- **Unused summaries**: Document summaries generated by LLM were stored but never used for search

### After
- **Semantic search**: Embedding vectors capture meaning and context
- **Better recall**: Query "배송비용" now matches "배송료", "shipping cost", "국제 배송 가격"
- **Summary-based**: Uses IndexEntry summaries for efficient search before loading full documents
- **Multilingual**: Korean and English queries both work effectively

### New Ports

#### EmbeddingGenerator
```typescript
interface EmbeddingGenerator {

  embedBatch(texts: readonly string[]): Promise<readonly (readonly number[])[]>;
  readonly modelId: string;
  readonly dimensions: number;

```

**Purpose**: Abstract text-to-vector transformation
**Implementation**: `LocalEmbeddingGenerator` using `@xenova/transformers`

#### EmbeddingStore
```typescript

  put(slug: string, vector: readonly number[], modelId: string): Promise<void>;
  get(slug: string): Promise<EmbeddingRecord | null>;
  getAll(): Promise<readonly EmbeddingRecord[]>;
  remove(slug: string): Promise<void>;

```

**Purpose**: Persist and retrieve embedding vectors
**Implementation**: `InMemoryEmbeddingStore` (MVP), `FileSystemEmbeddingStore` (Phase 2)

### New Services

#### SemanticIndexSearcher
```typescript
class SemanticIndexSearcher {
  async findTopK(queryVector: readonly number[], k: number): Promise<RankedSlug[]>
}
```

**Purpose**: Rank documents by cosine similarity
**Algorithm**: 
- Compute dot product between normalized query and document vectors
- Sort by descending similarity
- Filter out zero-similarity results
- Return top K slugs

### Updated Use Cases

#### AskAIUseCase
```typescript
constructor(
  documentRepository: DocumentRepository,
  questionAnswerer: QuestionAnswerer,
  embeddingGenerator?: EmbeddingGenerator,  // NEW: Optional
  indexSearcher?: SemanticIndexSearcher,    // NEW: Optional
)
```

**Changes**:
- Accept optional semantic search dependencies
- Try semantic search first if available
- Fallback to keyword search on failure or when not configured
- Zero impact on existing behavior when dependencies not provided

**Flow**:
1. User query → `embeddingGenerator.embed(query)`
2. Query vector → `indexSearcher.findTopK(vector, K=3)`
3. Top K slugs → `documentRepository.findById()` for each
4. Full documents → LLM context
5. LLM → Answer + sources

## Implementation Details

### Embedding Model

**Model**: `Xenova/multilingual-e5-small`
- **Dimensions**: 384
- **Languages**: Multilingual including Korean
- **Normalization**: L2-normalized vectors (cosine similarity = dot product)
- **Size**: ~120MB (downloaded on first use)
- **Provider**: Hugging Face via `@xenova/transformers`

### Why Local Model?

1. **No external API dependency**: Works without OpenRouter/OpenAI API keys
2. **No usage costs**: Free inference
3. **Privacy**: Document content never leaves local machine
4. **Korean support**: `multilingual-e5-small` trained on Korean text
5. **Consistent**: Local model file guarantees reproducible results

**Tradeoff**: Local model quality < OpenAI `text-embedding-3-small`, but sufficient for MVP and extensible via adapter pattern.

### Configuration

#### Environment Variables
```bash
# Enable semantic search (default: unset = keyword search)
EMBEDDING_PROVIDER=local
```

**Options**:
- `local`: Use `LocalEmbeddingGenerator` (multilingual-e5-small)
- `none` or unset: Use keyword search (backward compatible)
- Future: `openai` for OpenAI embeddings (adapter not yet implemented)

#### Composition Root

```typescript
const embeddingProvider = process.env.EMBEDDING_PROVIDER ?? 'local';
const embeddingGenerator = resolveEmbeddingGenerator(embeddingProvider, options);
const embeddingStore = new InMemoryEmbeddingStore();
const indexSearcher = embeddingGenerator 
  ? new SemanticIndexSearcher(embeddingStore) 
  : undefined;

const askAIUseCase = questionAnswerer 
  ? new AskAIUseCase(documentRepository, questionAnswerer, embeddingGenerator, indexSearcher)
  : null;
```

**Key design**: Optional dependencies + fallback ensure zero breaking changes.

## Performance Characteristics

### Embedding Generation
- **Cold start**: ~2-3s (model load)
- **Per document**: ~100-200ms (384-dim vector)
- **Batch**: More efficient via `embedBatch()`

### Search Performance
- **Query embedding**: ~100-200ms (one-time per query)
- **Top-K search**: O(N) where N = number of documents
  - 100 docs: <10ms
  - 1000 docs: ~50ms
  - For larger corpora, consider HNSW index (Phase 3)

### Memory Usage
- **Model**: ~300MB (loaded once, shared across requests)
- **Embeddings**: 384 floats × 4 bytes × N documents
  - 100 docs: ~150KB
  - 1000 docs: ~1.5MB

## Known Limitations

1. **Memory-only storage**: Embeddings lost on restart (Phase 2 will fix)
2. **No automatic indexing**: Existing documents require manual embedding generation via `scripts/build-embeddings.ts`
3. **Single model**: Only local multilingual-e5-small supported (extensible via adapter)
4. **No reranking**: Simple top-K, no hybrid scoring (Phase 3 candidate)
5. **Brute-force search**: Linear scan acceptable for small corpora (<1000 docs)
